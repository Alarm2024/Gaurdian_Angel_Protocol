Abstract

Guardian Angel Pro is a production‑grade asset rescue and recovery framework for Solidity smart contracts, developed and maintained by ELGHALY COMPANY. It provides protected, auditable paths for recovering ETH, ERC20, ERC721, and ERC1155 assets from contracts during security incidents, key compromise scenarios, or operational emergencies.

The protocol employs a defense‑in‑depth strategy through role‑based access control with strict privilege separation, timelocked governance for critical parameter changes, a modular architecture for auditability and extensibility, EIP‑712 typed signatures for cross‑chain replay protection, and real‑time execution limits to cap exposure. All components are self‑contained with zero external dependencies, making the full trust surface auditable in a single codebase.

1. Introduction

Smart contracts that hold significant value face a fundamental security challenge: they must be accessible enough to perform legitimate operations, yet secure enough to prevent unauthorised access. Traditional single‑key ownership models create dangerous single points of failure:

  • Key Loss — Permanent asset lockout if the owner's private key is lost
  • Key Compromise — Immediate theft of all assets if the owner's key is stolen
  • Governance Attacks — Malicious actors can exploit single‑party control
  • Cross‑Chain Replay — Signatures valid on one chain can be replayed on another

Guardian Angel Pro addresses these challenges through a multi‑role architecture that distributes authority across independent actors, combined with timelocked consensus mechanisms that prevent unilateral malicious actions. The protocol is designed to be self‑sovereign — the Owner retains ultimate control while being protected against their own potential mistakes or compromise.

2. Architecture

Guardian Angel Pro follows a modular, hook‑based design that separates concerns into three layers.

2.1 Design Principles

  • Modularity — Each module is responsible for a single concern, reducing audit complexity
  • Reusability — Core modules can be composed into any contract, not just GuardianAngel
  • Explicit Authorization — The _authorizeRescue() hook forces explicit policy definition
  • Extensibility — New policy modules can be added without rewriting underlying rescue transfers
  • Testability — Each module can be tested in isolation, ensuring correctness

3. Core Components

3.1 GuardianAngel

The complete implementation with all operational policies. This contract is the authority boundary in production: only the Owner configures the system, and only the Guardian executes emergency recovery flows.

Key Responsibilities:

  • Two‑step ownership management (transfer + accept)
  • Guardian rotation with 24‑hour timelock and dual‑party veto
  • Pause/unpause controls for circuit‑breaking
  • Chainlink Automation support via checkUpkeep and performUpkeep
  • ETH withdrawal paths: instant (cold signer) and timelocked (12h fallback)
  • Asset rescue entry points for ERC20, ERC721, and ERC1155
  • EIP‑2612 permit‑assisted ERC20 rescue for gas‑less user recovery

3.2 DailyLimitGuard

A reusable accounting module for daily spending limits per asset. Built around UTC day buckets using block.timestamp / 1 days.

Security & Gas Benefits:

  • No timestamp subtraction underflow — compares day buckets instead of subtracting timestamps
  • Timestamp manipulation impact is bounded — validator time drift only affects transactions near day boundaries
  • Cheaper resets — no separate reset transaction needed; the first operation in a new bucket directly writes the new bucket and amount
  • Zero limit = unlimited — provides flexibility for assets that should not be capped

3.3 RescueExecutor

A reusable transfer execution module that centralises all asset recovery logic. The module delegates policy decisions to hooks rather than hardcoding authorisation or rate‑limiting rules.

Supported Operations:

  • Native ETH transfers
  • ERC20 transfers (standard and permit‑based)
  • ERC721 transfers
  • ERC1155 transfers (single and batch)

Security Hooks:

  • _authorizeRescue(address to) — must be implemented to define who can rescue assets
  • _enforceDailyLimit(address asset, uint256 amount) — can be overridden to apply limits
  • All hooks are called before external transfers, preserving checks‑effects‑interactions

4. Module Interaction

4.1 Inheritance Pattern

Modular deployments are composed via inheritance and hook overrides:

contract MyProtocol is RescueExecutor, DailyLimitGuard {
    // Define who can rescue assets
    function _authorizeRescue(address to) internal view override {
        require(msg.sender == owner || msg.sender == guardian, "Unauthorized");
    }

    // Apply daily limits to all rescue operations
    function _enforceDailyLimit(address asset, uint256 amount)
        internal override(RescueExecutor, DailyLimitGuard)
    {
        DailyLimitGuard._enforceDailyLimit(asset, amount);
    }
}

4.2 Execution Flow

The interaction flow is strictly one‑way, preserving checks‑effects‑interactions:

  1. The caller enters a rescue function exposed by RescueExecutor
  2. RescueExecutor calls _authorizeRescue() before touching assets
  3. For value‑bearing paths, RescueExecutor calls _enforceDailyLimit(asset, amount)
  4. DailyLimitGuard accounts the current day bucket and reverts if the limit is exceeded
  5. Only after authorisation and accounting pass does RescueExecutor perform the external transfer

This ensures that state updates (nonce consumption, daily spending) always precede external calls, preventing reentrancy attacks and ensuring accurate accounting.

5. Security Model

5.1 Role‑Based Access Control

Guardian Angel Pro implements a four‑role authorisation model:

RoleCapabilitiesTrust Assumption
Owner Day‑to‑day operations: withdrawals, limit management, whitelist control, rotation initiation Hot wallet or multisig; more exposed but with safeguards
Guardian "Break glass" role: rescues, pause/emergency sweeps, veto power over rotations Independent key held by trusted party or institution
Cold Signer Offline key: solely authorises withdrawSecure() via EIP‑712 signatures Hardware wallet or air‑gapped device; minimal exposure
Recovery Address Destination address: receives all rescued and swept assets Secure cold storage or insurance fund

5.2 Key Security Mechanisms

  • Two‑Step Ownership Transfer — Prevents permanent lockout by requiring the new owner to accept ownership, eliminating the risk of transferring to an unreachable address.
  • 24‑Hour Timelocked Rotations — All role rotations require a 24‑hour delay and can be cancelled by either the Owner or Guardian during this window, preventing unilateral malicious changes.
  • EIP‑712 Typed Signatures — Signatures are bound to the specific contract address and chain ID, with a typed data structure including explicit fields, nonces, and deadlines.
  • 12‑Hour Timelocked Fallback — If the cold signer key is lost, the Owner can initiate a withdrawal that executes after 12 hours; the Guardian can cancel during this window.
  • Daily Withdrawal Limit — Prevents large‑scale asset extraction even with a valid signature, with whitelist bypass for trusted destinations.
  • Circuit Breaker — Pausing blocks all withdrawal paths and enables the Guardian's emergency sweep, providing a "break glass" option during security incidents.
  • Reentrancy Protection — All state‑mutating external calls are protected by a custom nonReentrant modifier enforcing checks‑effects‑interactions ordering.

8. Timelocked Governance

Guardian Angel Pro implements two distinct timelock mechanisms:

8.1 Role Rotation Timelock (24 hours)

  • Applied to Guardian, Recovery Address, and Cold Signer rotations
  • Provides a 24‑hour observation window for the other privileged role to cancel
  • Prevents unilateral role changes without dual‑party consent
  • Cancellation can be triggered by either Owner or Guardian during the window

8.2 Withdrawal Timelock (12 hours)

  • Applied to the signer‑independent fallback withdrawal path
  • Provides Guardian oversight for Owner‑initiated withdrawals
  • Balances Owner access with Guardian veto power
  • Ensures the Owner is never permanently locked out of contract funds

9. EIP‑712 Implementation

9.1 Domain Separator

_DOMAIN_SEPARATOR = keccak256(
    abi.encode(
        EIP712_DOMAIN_TYPEHASH,
        keccak256("GuardianAngel"),
        keccak256(bytes(VERSION)),
        block.chainid,
        address(this)
    )
);

9.2 Withdraw Type

Withdraw(address owner, uint256 amount, uint256 nonce, uint256 deadline)

9.3 Verification Flow

  1. Validate signature deadline
  2. Compute struct hash: keccak256(abi.encode(WITHDRAW_TYPEHASH, owner, amount, nonce, deadline))
  3. Compute digest: keccak256(abi.encodePacked("\x19\x01", _DOMAIN_SEPARATOR, structHash))
  4. Recover signer from digest and signature using ecrecover
  5. Verify recovered address matches cold signer

9.4 EIP‑5267 Compliance

The contract exposes eip712Domain() for wallet integration, allowing users to verify the signing context before approving transactions. This enables hardware wallets and dApps to render the request safely for the cold signer to review.

12. Conclusion

Guardian Angel Pro represents a comprehensive solution to the challenge of securing digital assets in smart contracts. Through its modular architecture, multi‑role authorisation, timelocked governance, and multiple independent security mechanisms, it provides enterprise‑grade protection against the most common attack vectors in DeFi and asset management.

The protocol's emphasis on defence in depth ensures that no single point of failure can lead to catastrophic asset loss. The separation of roles ensures that each actor has precisely defined, minimal necessary capabilities. The transparent, auditable nature of all operations enables effective incident response and governance.

As the DeFi ecosystem continues to evolve, Guardian Angel Pro provides a robust foundation for secure asset management that can adapt to emerging threats while maintaining the highest standards of security and transparency.

Key Takeaways:

  • Zero external dependencies — full trust surface auditable in a single codebase
  • Modular design — smaller audit surface, reusable components, easier to extend
  • Defence in depth — multiple independent security layers protect against key compromise
  • Timelocked governance — dual‑party veto prevents unilateral malicious actions
  • EIP‑712 signatures — domain‑separated, replay‑proof, cross‑chain safe
  • Asset rescue — supports ETH, ERC20, ERC721, and ERC1155 with permit‑based gas‑less rescue
